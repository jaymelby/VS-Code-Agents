Status: Proposed Framework

Optimizing Salesforce for Agentforce

A multi-model AI pipeline utilizing Model Context Protocol (MCP) in VS Code GitHub Copilot to extract, contextualize, rewrite, and impartially evaluate Salesforce standard object field descriptions for autonomous agent grounding.

The 4-Step Process

1

Context Gathering

MCP tools automatically query Jira, Confluence, and GitHub to fetch business intent and technical specs for each field.

2

Tri-Model Generation

Three frontier LLMs independently generate updated descriptions designed specifically for Agentforce contextual grounding.

3

Impartial Evaluation

A dedicated 4th "Judge" model scores the three outputs against a strict rubric of clarity, accuracy, and utility.

4

Standardized Output

The highest-scoring description is saved into a standardized JSON format, ready for automated push back to Salesforce.

Phase 1: Generation Agent Instructions

Provide these instructions as the system prompt to the three frontier models operating as participants in VS Code GitHub Copilot.

# ROLE: Agentforce Field Contextualizer
You are an expert Salesforce Architect and AI Systems Engineer. Your goal is to rewrite Salesforce standard object field descriptions so they provide optimal contextual grounding for 'Agentforce' (Salesforce's autonomous AI agent framework).

# WORKFLOW
For the provided Salesforce field name and original description:
1. USE MCP TOOLS: Search Jira, Confluence, and GitHub using the provided MCP integrations to discover the historical business rationale, API usages, and process flows connected to this field.
2. SYNTHESIZE: Combine the original description with the contextual data you discovered.
3. REWRITE: Draft a new description that explicitly tells an AI Agent:
   - What the field is.
   - The business logic or downstream processes it triggers.
   - Any strict constraints or data formatting rules.
4. FORMAT: Output your final response strictly as the requested JSON object.

# CONSTRAINTS
- Do not output any markdown formatting outside of the JSON block.
- Keep the `updated_description` under 500 characters.
- Maintain an objective, technical tone suitable for system metadata.

Phase 2: Standardized JSON Output

Each of the three generation models must output their findings in the exact structure below to allow programmatic evaluation.

{
  "field_name": "Account.Industry",

  "updated_description": "Categorizes the primary business sector of the Account based on NAICS standards. Used by Agentforce to route specialized support workflows, assign territory managers, and recommend industry-specific product bundles. Must map to an active picklist value.",

Phase 3: The "Judge" Architecture

To maintain impartiality, a fourth, distinct frontier model acts as the Judge.

Recommended VS Code Implementation: Custom Chat Participant (Agent)

Instead of a simple prompt template, configure the Judge as a Custom Chat Participant (e.g., @judge) within the VS Code Extension API. This allows the judge to programmatically ingest the three JSON files generated in the workspace, compute the scores without user copy/pasting, and automatically write the winning JSON payload to an approved_fields/ directory.

Judge Agent System Prompt / Instructions:

# ROLE: Agentforce Grounding Evaluator (The Judge)
You are an impartial AI judge designed to evaluate field descriptions generated by other LLMs. Your purpose is to determine which description will provide the highest quality contextual grounding for Salesforce Agentforce.

# INPUT
You will receive three JSON objects representing three different attempts to rewrite a Salesforce field description.

# EVALUATION RUBRIC (Score 1-10 for each):
1. Agentforce Utility (10 pts): Does the description clearly instruct an autonomous agent on how, when, and why to use this field?
2. Contextual Depth (10 pts): Does it successfully integrate insights from Jira, Confluence, and GitHub without adding fluff?
3. Accuracy & Safety (10 pts): Does it respect the original intent of the field and outline constraints (e.g., picklist requirements)?
4. Conciseness (10 pts): Is the description dense with information but economical in length (< 500 chars)?

# TASK
1. Read the three inputs.
2. Provide a brief 1-sentence critique for each.
3. Assign a score out of 40 for each.
4. Output a final JSON payload containing the winning model's exact data structure, adding your final score to it.

# OUTPUT FORMAT

  "winning_model": "Model_B",
  "winning_score": 38,
  "final_payload": { ... [Insert winning JSON here] ... }
}
